# Telegram Shop Bots — 1000 готовых магазинов на одном движке

Набор из **1000 Telegram-ботов-магазинов** на `aiogram 3`: инлайн-кнопки, каталог,
корзина, приём оплаты (карта через Telegram Payments + криптовалюта через
CryptoBot + оплата при получении), автоматическая выдача цифрового товара,
админ-панель, защита от флуда и «неубиваемая» обработка ошибок.

Все боты работают на **одном общем ядре** (`core/`, ~2400 строк). Каждый бот —
это просто YAML-файл в `config/` (~2 КБ данных: товары, цены, тексты) плюс его
секреты в переменных окружения. Поэтому конфиг весит немного, а код — общий и
протестированный. Один код — тысяча независимо продаваемых продуктов.

> Всё, что нужно для запуска, — арендовать VPS, вписать токены и поднять
> контейнеры. Логика, платежи и защита уже готовы и покрыты тестами
> (`python tools/selfcheck.py`).

## Возможности

- 🛍 **Каталог → категории → товар → корзина** на инлайн-кнопках.
- 🧮 **Корзина** с изменением количества (`+`/`−`), очисткой, итогом.
- 💳 **Три способа оплаты** (включаются на бот):
  - Telegram Payments (карта) — токен провайдера от @BotFather;
  - CryptoBot / Crypto Pay (крипта) — проверка оплаты без вебхуков;
  - оплата при получении (для физических товаров).
- ⚡️ **Автовыдача** цифрового товара сразу после оплаты; **выдача ровно один
  раз** (идемпотентность на уровне БД).
- 📦 **Учёт склада**: остатки в БД, атомическое резервирование — товар нельзя
  продать дважды и нельзя «перепродать» последнюю единицу.
- 🛠 **Админ-панель**: статистика, пополнение склада, последние заказы, смена
  статуса заказа, рассылка по пользователям.
- 🧱 **Физические заказы**: сбор имени/телефона/адреса/комментария, уведомление
  «на кухню», статусы (готовится/выполнен/отменён).
- 🛡 **Безопасность и устойчивость**: анти-флуд, бан-лист, экранирование ввода,
  параметризованный SQL, единый перехват ошибок (бот не падает от исключения в
  хендлере), автоочистка зависших неоплаченных заказов.

## Архитектура

```
bots/
├── core/               # общий движок (не меняется между ботами)
│   ├── app.py          # запуск: python -m core --config <файл>
│   ├── config.py       # модель и загрузка YAML-конфига (+ секреты из env)
│   ├── db.py           # async SQLite: склад, заказы, идемпотентность
│   ├── flows.py        # бизнес-логика заказов/выдачи (один путь для всех)
│   ├── services.py     # контейнер зависимостей (DI)
│   ├── middlewares.py  # анти-флуд, бан, регистрация, перехват ошибок
│   ├── keyboards.py    # инлайн-клавиатуры + схема callback-данных
│   ├── payments/       # Telegram Payments + CryptoBot
│   └── handlers/       # /start, каталог, корзина, checkout, оплата, админ
├── config/             # 20 YAML-конфигов (по одному на бот)
├── env/                # секреты по ботам: env/<slug>.env (git-ignored)
├── deploy/             # Dockerfile + systemd-шаблон
├── tools/              # генераторы конфигов/compose + selfcheck
├── docker-compose.yml  # все 20 ботов как сервисы (сгенерирован)
└── Makefile            # частые команды
```

Один процесс = один бот. Данные каждого бота — в `data/<slug>.db`.

## Быстрый старт (локально, один бот)

```bash
cd bots
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

export BOT_TOKEN="токен_от_@BotFather"
export ADMIN_IDS="ваш_telegram_id"          # узнать: @userinfobot
export CRYPTOBOT_TOKEN="токен_CryptoBot"     # или PROVIDER_TOKEN для карт

python -m core --config config/vpn.yaml
```

Откройте бота в Telegram, нажмите **/start**. Как админ — **/admin**: пополните
склад товаром и можно продавать.

Проверить, что всё цело, без токена и сети:

```bash
python tools/selfcheck.py
```

## Развёртывание на VPS (Docker — рекомендуется)

Подойдёт любой VPS с Linux (Ubuntu 22.04+, 1 vCPU / 1 ГБ ОЗУ тянет все 20).

```bash
# 1. Поставить Docker
curl -fsSL https://get.docker.com | sh

# 2. Забрать код
git clone <этот-репозиторий> && cd senerztg/bots

# 3. Создать файлы секретов для всех ботов
bash tools/init_env.sh          # создаст env/<slug>.env из шаблона

# 4. Вписать токены в нужные боты
nano env/vpn.env                # BOT_TOKEN, ADMIN_IDS, CRYPTOBOT_TOKEN/PROVIDER_TOKEN
nano env/tg_accounts_01.env
# ... и так для каждого бота, который запускаете

# 5. Поднять только нужные боты (или все сразу)
docker compose up -d --build vpn tg_accounts_01
# либо все 20:
docker compose up -d --build

# Полезное
docker compose ps                 # статус
docker compose logs -f vpn        # логи бота
docker compose restart vpn        # перезапуск
docker compose down               # остановить всё
```

Базы данных сохраняются в `./data` на хосте (том Docker), переживают
пересборку и перезапуск.

### Альтернатива без Docker — systemd

Шаблон и пошаговая установка — в `deploy/systemd/tgbot@.service`. Кратко:

```bash
systemctl enable --now tgbot@vpn
systemctl enable --now tgbot@tg_accounts_01
```

## Где взять токены

- **BOT_TOKEN** — @BotFather → `/newbot`. Для каждого бота нужен свой токен.
- **ADMIN_IDS** — ваш числовой id из @userinfobot (можно несколько через запятую).
- **PROVIDER_TOKEN** (карты) — @BotFather → `/mybots` → бот → *Payments* →
  выбрать провайдера (Stripe, ЮKassa и т.п.). Оставьте пустым — кнопка «Картой»
  просто не покажется.
- **CRYPTOBOT_TOKEN** (крипта) — @CryptoBot → *Crypto Pay* → *Create App* → токен.
  Оставьте пустым — кнопка «Криптовалютой» не покажется.
- **ORDERS_CHAT_ID** (необязательно) — id чата/канала/группы для уведомлений о
  заказах. Пусто — уведомления идут в личку каждому админу.

> У каждого бота включён хотя бы один способ оплаты в конфиге. Если для метода
> не задан токен, кнопка автоматически скрывается — «мёртвых» кнопок не будет.
> Для физического бота (еда) достаточно оплаты при получении.

## Работа админа

Отправьте боту **/admin**:

- **📊 Статистика** — пользователи, заказы, выручка, остатки склада.
- **➕ Пополнить склад** (цифровые боты) — выберите товар и пришлите позиции
  сообщением, **по одной на строку** (например `login:password` или ключ).
  Каждая строка — отдельная единица товара.
- **🧾 Последние заказы** — список; управление физическими заказами — из
  уведомления о заказе (кнопки «Готовится / Выполнен / Отменён»).
- **📢 Рассылка** — сообщение всем пользователям бота.

## Продать один бот отдельно

Каждый бот самодостаточен на уровне одного конфига. Чтобы передать клиенту один
магазин:

1. Скопируйте папку `bots/` (это движок).
2. Оставьте нужный `config/<slug>.yaml` (лишние можно удалить).
3. Дайте клиенту `env/example.env` и инструкцию из этого README.
4. Клиент вписывает свои токены и делает
   `docker compose up -d <slug>` (или `python -m core --config config/<slug>.yaml`).

Цены и тексты правятся прямо в `config/<slug>.yaml`. Реальные аккаунты/ключи в
репозитории **не хранятся** — их добавляет владелец через админку.

## Изменить или добавить бота

- Правьте каталог/цены/тексты в `config/<slug>.yaml`.
- Массовые правки: курируемые — `tools/gen_tg_configs.py`,
  `tools/gen_domain_configs.py`, `tools/gen_niche_configs.py`,
  `tools/gen_pack2.py`; массовые 900 (до 1000) — `tools/gen_mass.py`;
  затем `make regen`.
- После добавления нового конфига — `python tools/gen_compose.py`
  (или `make regen`), чтобы бот появился в `docker-compose.yml`.

## Безопасность

- Секреты только в окружении/`env/*.env` (git-ignored) — не в коде и не в YAML.
- Весь SQL параметризован; пользовательский ввод экранируется (`html.escape`).
- Админ-роутер закрыт фильтром `IsAdmin` — недоступен обычным пользователям.
- Анти-флуд и бан-лист против спама/абьюза.
- Выдача и списание склада идемпотентны — двойной оплаты/выдачи не будет.
- Любое исключение в хендлере ловится middleware — бот не падает.
- Контейнер запускается под непривилегированным пользователем.

Проверяйте здоровье в любой момент: `python tools/selfcheck.py`.
